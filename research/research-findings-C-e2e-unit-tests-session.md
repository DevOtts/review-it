## 2. QA methodology actually used

**Sequencing:** Read delivery package in mandated order (KICKOFF → README → CONTRACT → program-plan/epic) → play back understanding → build ONE slice (E3) fully end-to-end → **stop for human review** → get "approved, continue" → fan out remaining epics in parallel via background subagents → conductor (main session) verifies + merges each as it lands → later epics (E7/E6) done after a computer restart, resumed from committed state files.

**Per-epic loop (the pattern that repeated every time):**
1. Re-read the LIVE `.github/workflows/*` for the target repo before touching anything (EXTEND-NEVER-CLOBBER gate).
2. Confirm the test suite actually runs locally first (`npx jest --ci`), fixing any config issues discovered (e.g., master-admin's duplicate `jest` key in `package.json` shadowing `jest.config.js`).
3. Branch off `main` (`epic/qa-<id>-<repo>`), add/extend a workflow file wiring only scripts confirmed to exist (dead `test:e2e`/`test:integration` never wired).
4. Open PR, watch baseline CI run green.
5. **Adversarial red→green proof**: sabotage production code (not the spec) with a marked commit ("GATE PROOF — deliberate regression, WILL BE REVERTED"), watch the gate go red and block, `git revert`, watch it go green again. Captured exact CI run IDs for every phase (e.g., master-admin: baseline `28765935929` ✅ → sabotage `28766012428` ❌ → revert `28766071062` ✅).
6. Leave subagent-built PRs **open**; the conductor independently re-verifies (byte-diff untouched workflows, re-check CI, spot-check row counts/claims) before merging centrally.
7. Update a single owned `STATUS.md` board and `TaskList` after each merge; reap subagents/worktrees once merged and captured.

**Subagent orchestration:** parallel `Agent` calls (`run_in_background: true`) fanned out E1/E2/E4/E5 simultaneously, model-tiered: haiku for mechanical gate-wiring (E2, E4), sonnet for spec'd build work (E1 catalog, E5 deploy-gate, E6 integration lane), fable/conductor-direct for the hardest cross-cutting reasoning (E7 contract amendment). Conflict guards: catalog file was E1-exclusive (others reported suite inventories back for reconciliation instead of writing it); STATUS.md stayed conductor-owned; parallel repo work used `git worktree` isolation (`<repo>/.worktrees/<name>/`, per RD-8) so the main checkout stayed untouched.

**Test-contract discipline:** every epic had a fixed ≥10-case Test Contract (e2e / adversarial / grep-proof / unit categories) that had to hit 100% before "shipped"; any case blocked by an unreachable real dependency was marked `IMPLEMENTED-NOT-VERIFIED`, never asserted PASS.

## 4. Right decisions worth keeping

- **Ship-one-slice-then-stop-for-review** before fanning out — proved the entire catalog→gate→block loop end-to-end on the cheapest repo (master-admin) before parallelizing, catching the jest-config gotcha early and cheaply.
- **Adversarial red→green proof as the actual acceptance test for "a gate exists,"** not just "CI is green": sabotage production code (never the spec), capture the failing run ID, revert, capture the passing run ID. This was applied uniformly across E3/E2/E4/E5/E6/E7's adversarial controls (e.g., E7's SYN-1 synthetic-invariant control, tampered `fp9-9.99` mapping control).
- **No-fake-green discipline, applied bidirectionally**: not only "don't claim PASS for an unreachable case" (E2's 41-failure baseline, E6's TC-8/TC-9 → honestly `IMPLEMENTED-NOT-VERIFIED` with a stated blocker), but also "don't leave something INV if it's actually now verifiable" — when Docker turned out to be running, the conductor had E6 convert 8 pre-marked-INV cases into real live PASS instead of leaving the easy/pessimistic default.
- **EXTEND-NEVER-CLOBBER verified by byte-diff, not by trust**: e.g. explicit `git diff main -- .github/workflows/docker-build.yml` showing zero hunks before merging E2/E6; independent verification that `needs: [lint, test]` was the only structural change.
- **Independent conductor re-verification of every subagent's claims before merge** — counting catalog rows itself (`grep -c`), re-running failing specs, diffing spec files for "additive-only" claims — rather than merging on a subagent's self-report.
- **Worktree isolation + `.env` copy convention (RD-8)** kept 4 parallel epic-builder agents from colliding on the same repos, and made cleanup ("remove worktree, prune, delete branch") a clean, repeatable step.
- **Model tiering matched to task difficulty**: haiku for mechanical CI-wiring epics, sonnet for spec'd builds, fable/conductor-direct for the CONTRACT-amendment reasoning (E7) — with an explicit remediation escalation (spawn a dedicated sonnet remediation agent) when a haiku-tier gate epic hit unexpected complexity (41 pre-existing failures).
- **Durable state before risk (computer restart)**: before an intentional reboot, the session committed everything, verified 0 uncommitted/0 ahead-behind across every touched repo, listed exactly what belonged to *other* concurrent sessions and left it untouched, and left a `NEXT-SESSION-PROMPT.txt` + `Task #6` so the resumed session picked up with zero re-derivation.
- **Respecting concurrent sessions sharing the same working directory**: repeatedly checked which branch/worktree belonged to another live session (ontology session's brain-docs WIP, master-admin's other epic branch, `.plan-it/*.state.json`) and used additive, delete-nothing edits (e.g. prepending to `hot.md` anchored on the current top line) rather than blind overwrites; deliberately skipped committing to a repo (vault, brain-docs) when doing so risked sweeping up someone else's uncommitted work.
- **Test-suite-rot-as-a-feature, not a bug**: when a new gate exposed pre-existing failures (E2's 41 failures, E6's `graph.service.ts` DI bug), the response was root-cause remediation or an explicit out-of-scope handoff — never suppressing/skip-listing the failing tests to force green.
- **Catalog built programmatically, not hand-transcribed**: E1's 708-row catalog was produced by a Python script parsing the three harvest source tables directly, guaranteeing exact counts / zero ID collisions / zero fabricated content at scale — validated by a 12-case self-check (TC1-01..12) including an adversarial delete-one-row-and-confirm-count-drops check.

## 5. Skill-design implications for the new test-it / review-it / qa-it skill

- **Encode a mandatory pre-flight step**: before adding/editing any CI workflow, `cat` the repo's *live* `.github/workflows/*` in full and diff after — never trust memory of what a workflow contains. Treat unchanged-workflow byte-diff as a hard gate check, not a nice-to-have.
- **Encode "wire only scripts that actually resolve"**: grep `package.json` scripts and confirm the target directories/files exist before wiring a gate to them; dead scripts (`test:e2e`, `test:integration` pointing nowhere) are never gate targets.
- **Make the red→green adversarial proof a required, not optional, step of "done"** for any new/edited gate: sabotage prod code with a clearly-labeled revert-me commit, capture the failing run ID, revert, capture the passing run ID — and require both IDs in the completion report.
- **Bake in a no-fake-green vocabulary**: three explicit states — PASS, FAIL, `IMPLEMENTED-NOT-VERIFIED` (with a one-line blocker reason) — and forbid collapsing the third into the first. Also require re-checking previously-INV cases whenever new capability appears (e.g., Docker available) rather than leaving them stale.
- **Require CI (remote/merged truth) as the authoritative verification surface, not local runs** — local repos are contaminated by other sessions' worktrees, stale branches, and corrupted `node_modules` in ways CI's clean checkout is not. Any claim about "current" catalog/coverage state should be checked against `origin/main`, never a local checkout that might be parked on a stale/foreign branch.
- **Concurrency hygiene as a first-class rule**: before writing to any shared file/repo (STATUS boards, vault `hot.md`, shared docs repos like brain-docs), check for other sessions' uncommitted/unpushed work first; prefer additive anchored edits over destructive rewrites; skip committing entirely if the local working tree holds someone else's WIP.
- **Subagent self-reports are not verification** — the coordinator (or the skill's orchestration layer) must independently re-derive load-bearing claims (row counts, byte-diffs, CI conclusions) before treating an epic/PR as ready to merge. Treat a subagent's first-pass "green"/"done" claim as provisional until independently checked.
- **One-slice-then-review before fan-out** is a good default gate for a new QA methodology: prove the full loop (catalog → gate → red → green → merge) on the cheapest/smallest target first, get explicit human sign-off, then parallelize the same proven pattern.
- **Standardize the completion report shape** used across every epic here — deliverable links, adversarial-proof table (commit/run-id/result), Test Contract case table with per-case PASS/FAIL/INV, contract-compliance checklist, gotchas — this format made cross-epic verification fast and should be templated directly into the skill.
- **Persist resumable state before any long/interruptible run** (STATUS.md board, NEXT-SESSION-PROMPT.txt, TaskList) so a restart or context-compaction loses zero context — this matches the user's global overnight-resilience instructions and should be a built-in checkpoint step of the skill, not an ad hoc habit.
- **Model-tier the fan-out explicitly and record tiering + escalations** (haiku for mechanical CI wiring, sonnet for spec'd builds/remediation, fable/conductor for contract-level reasoning) as part of the plan output, with an explicit escalation path when a lower tier thrashes (as happened with E2's 41-failure remediation).
