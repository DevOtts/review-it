## 2. Root cause of the 3 flagged failures

### Failure 1 — Credentials directive: **not found in this transcript as described**
I grepped exhaustively (`env var`, `k8s`, `kubectl set env`, `AIRTABLE_PAT`, "should I/want me to" near credentials, every subagent's Agent-prompt, every teammate-message) and could not find a moment where this session asked to add the Airtable PAT (or any related secret) as a k8s env var. What the transcript actually shows is the opposite pattern followed correctly throughout: the PAT is pulled via `.secrets/.full.credentials` → the `airtable` Integration doc, using a "multi-level credential path walker" in every `.real.spec.ts` file (explicitly called out at line ~1186 and ~1195), and E6's own report states outright: *"No secrets added, no new env vars, nothing touched outside the settings-page slice."* E5 (brain-agent) explicitly reused three already-set prod env vars for unrelated config (`BRAIN_API_CORE_URL`, `SERVICE_API_KEY`, `OPENROUTER_API_KEY`) and stated *"No new env var"* — never proposed one for Airtable auth.
**My conclusion:** either this exchange happened in a different/parallel session (the concurrent brain-admin-ui session, or the earlier planning session that froze the CONTRACT) or in a part of this conversation the distillation compressed out. I'm flagging this rather than inventing a quote — worth checking the sibling session transcript if you want the exact moment. The gate recommended in §5 stands regardless of where it happened.

### Failure 2 — False e2e pass via CDP: **confirmed, root cause is real**
At the E7 rollout-gate report (line ~2195, ~2215, ~2254) the session declares: *"That's the full E7 flow verified short of the actual write, which is exactly the rollout gate,"* and later, in the coordinator report: *"the modal renders 13 buttons on the real Benari meeting, prefill returns an LLM-rewritten title/description with the project suggestion pre-selected, and Create correctly shows a real inline 403 while writes are off (no fake green)."*
What the CDP script actually exercised: (1) button count on the renderer, (2) that prefill populated title/description/project fields, (3) that clicking Create with writes off produced the expected 403 and kept the modal open. **It never interacted with the comboboxes as controls** — no typing into the search input, no opening/closing the popover, no clicking a non-default option, no checking whether an empty control loaded at all. Those are exactly the 4 bugs Fernando found by hand:
- can't type in search (client-side filter fighting server-driven options)
- clicking a dropdown option opened the field *behind* it (Radix `pointer-events:none` portal bug)
- Assignee never loaded/pre-filled
- Project dropdown locked to one option, no search

The session's own root-cause note confirms the CDP pass never exercised this surface: *"Real UX bugs in the modal — let me diagnose the combobox behavior before fixing… guessing at Radix stacking wastes time"* — i.e., it had to reproduce all of this live *after* the human report, meaning it had never been reproduced before. This is a **"verified the wrong layer"** failure: the CDP pass asserted on rendered *text* (did fields populate) and a *terminal-state* response code (403), but never asserted on interactive *state transitions* of the controls a user would actually operate.

### Failure 3 — Unverified third-party write: **confirmed**
Per CONTRACT D-W4, the actual prod `enableWrites` flip + create was reserved for Fernando ("the single human-reserved, outward-facing decision"), and the session repeatedly *offered* to run the sanctioned verification afterward: *"tell me and I'll run the sanctioned #W4 verification (create → 'See in Airtable' persists across reload + re-ingest)"* (line ~2263, ~2286, ~2603). But Fernando flipped the toggle and clicked Create himself before handing control back — so the very first live create-and-look-in-Airtable was done by Fernando, not the session, and he's the one who caught it was empty (line ~2606‑2608: *"it show me an empty task"*).

Digging one layer deeper: even before this, none of E1–E4's `[REAL]` tests would have caught this class of bug, because every one of them wrote to the scratch table `test_lvis` and verified success purely via **API field-value GETs** (curl), never by opening the Airtable UI page and looking at how the record renders. The actual defect — the `tasks` table's primary field (`task_id`) was never populated, so Airtable showed the record as "Start typing…" with computed fields floating to the top — is a *display/UI-semantics* bug that is invisible to a raw `GET .../records/{id}` field check (all fields WERE present and correct; see line ~2622‑2625: *"the record is **not** actually empty — the write worked perfectly… The problem is the primary field"*). So: **yes, the session did programmatically write+read-back records against a scratch table repeatedly, but never against the actual `tasks` table with its real primary-field constraint, and never by rendering the Airtable UI itself** — which is precisely the gap that let this ship.

Bonus finding in the same stretch: after the fix, the session's own re-verification hit a **transient false negative** (a second created record read back as fully null moments after creation, then fully populated on 3 stable re-reads) — Airtable's read-after-write lag. The session correctly didn't panic-report this as a regression, but it's a second illustration that "read the API back once" is not a sufficient verification primitive for this integration; it needs to be a stable, multi-read check by default, not just a break-glass reaction to a scare.

2. **The CONTRACT's own numeric error propagated uncaught for a while.** Line ~479: *"the original list actually has **12** entries (the docs' '11' was off by one)"* — a binding-law document (CONTRACT.md v1.2) had a wrong count that nobody had verified against the actual code before freezing it; only surfaced when the session went to edit it.

4. **Debrief captured tactical lessons, not the methodology lesson.** The four vault lessons appended (pointer-events-in-modal-Dialog, Airtable primary-field rule, Airtable REST quirks, brain-agent id resolution) are all real and useful, but none of them says "assert on interactive control state during CDP verification" or "read a write back through the actual third-party UI, not just its API," even though the session had just lived through both failures. This is a missed opportunity to fix the *process* rather than just the *bugs*.
5. **The E5 subagent independently caught two real bugs before shipping** (a flattened-vs-nested record shape assumption, and the LLM echoing the input title verbatim) — this is *good* behavior worth noting as a positive counter-example: it happened because that epic's Test Contract had `[REAL]` cases that forced a live run against prod, not a mock. ## 5. Concrete gates for a new QA skill

1. **Write-verification gate (closes Failure 3 / pattern A, C):** A write to a third-party system is never `VERIFIED` until the record has been **read back from that system's own surface that a human will actually look at** — for a system with a UI (Airtable, a CRM, etc.), that means loading the record's page/view, not just calling its read API. If the write target has both an API and a UI, verification requires both: API GET for field-level correctness, UI load for display/primary-field/rendering correctness. A GET-only check is `IMPLEMENTED-NOT-VERIFIED-IN-UI`, not `VERIFIED`.
2. **Read-back stability gate:** Any read-after-write check that returns an unexpected result (null, stale) must be re-read at least 2–3 times before either (a) declaring a real defect or (b) declaring VERIFIED — a single read is not evidence, since some third-party stores (Airtable) have read-after-write lag. Log the raw sequence of reads as evidence, not just the final one.
3. **Interactive-control coverage gate (closes Failure 2 / pattern B):** A CDP/e2e pass over a form/modal must, for **every interactive control touched by the DoD**, assert on at least: (a) initial rendered state, (b) the primary user action for that control type (type into a search box, open a popover, select a non-default/non-preselected option), (c) the resulting state after that action. "The field shows a value" is not sufficient evidence that "the field is searchable/editable" — those are different assertions and the DoD/Test Contract should list them as separate cases whenever the control supports both a display state and an interaction state.
4. **First-look-before-handoff gate (closes Failure 3 / pattern C):** When an autonomous run reserves an action for the human (e.g., "you flip the switch"), and the run also commits to running its own verification right after — that commitment should include an explicit "don't act on the reserved action without pinging me first" ask, or the run should race to observe the very first live state itself (e.g., poll for the flag flip) rather than assuming it'll get first look. Otherwise the human ends up doing the first real-world test, un-instrumented.
5. **Standing-directive lookup gate (closes Failure 1 / pattern D):** Before proposing *any* new credential storage location (env var, k8s secret, `.env` file, hardcoded config), grep the frozen CONTRACT/decisions file and the kickoff message for existing rulings on that exact question, and quote the ruling back instead of asking. If no ruling exists, default to the existing credential-store pattern already used elsewhere in the codebase rather than introducing a second mechanism.
6. **Deploy-safety gate (closes pattern E):** Script the merge→build→deploy step so a squash-merge-with-delete-branch cannot leave the working directory checked out on `main` with an uncommitted diff ready to push — explicitly `git checkout -b <fix-branch>` before any edit, and fail loudly if a `git push` target resolves to `origin/main`.
7. **Debrief methodology gate (closes pattern F):** The session-debrief skill should explicitly prompt: "did any DoD item get a false VERIFIED this session, and if so, what verification primitive would have caught it earlier?" — not just "what bugs did we fix" — so the lesson captured is about the *test*, not only the *code*.
